# 静态库

## 说明

静态库是**在程序链接时被包含进可执行文件**的库。它的扩展名通常是`.a`或`.lib`。

优点：可执行文件包含了所有需要的代码，因此可以独立运行，不依赖外部库。

缺点：可执行文件大小更大，库更新后需要重新编译整个程序。

## 实现

- 编译目标文件：先用`gcc -c`把需要的`.c`变成`.o`

- 打包：`ar -cr libxx.a yyy.o`，将所有的`.o`目标文件打包成库，`xx`为库名。

- 头文件发布到：`/usr/local/include`；库发布到：`/usr/local/lib`。以上是`gcc`编译时默认的搜索位置。

- 使用：编译时链接上对应的静态库（静态链接），即`gcc main.c -o main -lxx`。依赖多个库时，被依赖的写后面。如果头文件和库没有发布到上述位置，也可加上` -I -L`指定编译时搜索头文件和库的位置。

此时库就已经被塞到生成的可执行文件中了。

# 动态库

## 说明

动态库在程序编译时不会被包含进可执行文件，只会在可执行文件中包含需要加载的库名和符号信息，而是**在程序运行时被加载**。动态库的文件扩展名通常是 `.so`或` .dll`。

优点：减少了内存占用，因为多个程序可以共享同一份库代码；更新库文件时不需要重新编译使用它的程序；还可以在程序运行时按需加载库。

缺点：加载速度比静态库慢，且分发程序时需要确保动态库的可用性。

## 实现

- 编译出位置无关的目标文件`.o`并打包为库：`gcc -shared -fpic -o libxx.so yyy.c`
- 头文件发布到：`/usr/local/include`；库发布到：`/usr/local/lib`。以上是`gcc`编译时默认的搜索位置。
- 使用：编译时链接库（动态链接）`gcc main.c -o main -lxx`。同样可以使用`-I -L`，但是这只是告诉了编译器库的位置。
- 运行：由于动态库是在运行时加载的，因此此时还要告诉操作系统库在哪。

​	对于非root用户：声明环境变量`export LD_LIBRARY_PATH=库具体路径`
