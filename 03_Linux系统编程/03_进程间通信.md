# 进程间通信方法

进程之间的内存是隔离的，如果多个进程之间需要进行信息交换，有以下几种常用方法：Unix Domain Socket IPC、管道（有名管道、无名管道）、共享内存、消息队列、信号量。

# 管道

单向的进程间通信方式，直接使用内核。

## 匿名管道Pipe

### 系统调用原型

`int pipe(int pipefd[2])`

`pipefd[0]`（读端）：从这个文件描述符里读取数据。

`pipefd[1]`（写端）：向这个文件描述符写入数据。

### 功能机制

- 匿名管道是内核内存中一块由环形缓冲区管理的临时存储区域。它没有名字没有文件路径，只以一对用于读写的文件描述符的形式存在。由于没有名字所以只能用于有亲缘关系的进程。
- 普通文件描述符需要通过`open`先找到或创建`struct file`，然后返回一个`fd`。但是管道在创建时就直接创建了`struct file`并返回了一对`fd`，所以管道不需要也不能`open`。
- 使用时，在`fork`之前创建管道（即一对文件描述符），`fork`后子进程也有了一对，并且和父进程指向同一个`struct file`。所以可以在一个进程写数据到写端，另一个进程从读端读数据。
- 写数据的那个进程必须要立即关闭读端，并且写完要关闭写端，这是为了告诉内核这一端用完了，不会有数据进来了，不用等了，从而触发另一进程读端的EOF。读数据的那个进程同理相反。
- 管道是单向的，一旦确定了某一方是发，那这一端在这根管道上永远只能发，否则会出现通信混乱。

## 有名管道FIFO

### 系统调用原型

`int mkfifo(char *path, mode_t)`

参数1为管道绑定的文件路径，参数2为管道绑定文件的权限

### 功能机制

- 有名管道能让任意两个进程互相通信，它绑定一个文件名，但是数据依然是从内核的环形缓冲区走。
- 想要用有名管道读写，都需要`open`获得`fd`，两个进程都获取到这个管道的`fd`时就能进行通信了。
- 管道只需要在其中一个进程创建一次，并且要在这个进程最后用`unlink()`释放管道。
- 有名管道也是单向的，所以写数据的进程`open`时给只写权限，另一进程给只读权限。

# 共享内存

速度最快、效率最高的进程间通信方式。直接在物理内存中开辟一块位置，让两个进程的虚拟地址同时映射到这块位置上。两个进程就可通过修改或访问这块内存上的数据来进行间接通信。不推荐使用一个共享内存进行双向通信

## 相关系统调用

### shm_open

`int shm_open(char *, int, mode_t)`

参数1为/文件名；参数2为打开模式；参数3为权限；返回值为文件描述符

- 通过该函数可以以文件的方式创建并打开一个的内存块
- 共享内存对象不是普通文件，但它像文件一样拥有一个路径名（必须斜杠 `/` 开头）和一个文件描述符（`fd`）,这个路径名只是一个内核用来查找对象的标签，不是硬盘上的文件路径。
- `shm_open`创建的对象默认大小为0。

`shm_unlink(char *)`：

参数为要删除的文件名，即`shm_open`中创建的。

- `shm_unlink`用于删除先前创建的命名共享内存对象，但没有真正删除共享内存段本身，而是取消关联了文件名，当所有进程都去取消关联了，系统才会释放。
- 只需要创建共享内存对象的进程执行它去删除

### truncate

`int truncate(char *, off_t length)`

`int ftruncate(int, off_t)`

参数1目标对象；参数2为设定的最终大小（字节数）

- 核心作用时把文件（或共享内存对象）的大小切成指定的长度
- 如果原文件大于`length`，内核会丢弃超出的末尾部分；如果原文件小于`length`，内核会通过`\0`扩展到指定位置

### mmap

`void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t);`

参数1为指定映射的起始虚拟地址；参数2为要映射的字节数；参数3为内存保护权限；参数4为修改内存的逻辑；参数5为共享内存的文件描述符；参数6为开始映射的位置；成功返回映射区域的起始地址

- `mmap`可以把文件、设备或共享内存对象映射到进程的虚拟地址空间中，映射后就可以通过返回的指针像操作数组一样访问内存。
- 映射后要`close(fd)`，因为不需要通过fd读写了。但是fd引用计数不会减少
- `addr`填`NULL`，内核会自动选择一个合适的起始地址
- `length`不能超过共享内存大小。
- `flags`一般选`MAP_SHARED`，即对内存的修改会立刻同步到共享内存上。

`int munmap(void *addr, size_t length);`

参数1为`mmap`返回的指针；参数2为映射时的大小

- 不需要这块共享内存时（比如进程要退出），用来解除映射。
- 有进行映射的进程都要在最后解除映射，因为`mmap`是把内存映射到自己的进程上。
- 只是解除了当前进程对物理内存的映射，没有直接释放内存

## 实现流程

1. 使用`shm_open`创建共享内存对象
2. 使用`truncate`设置共享内存对象大小
3. 使用`mmap`内存映射获取返回的地址，映射后关闭fd
4. 后续操作`mmap`返回的地址实现进程间通信
5. 双方都释放映射区`munmap`
6. 创建方释放共享内存对象`shm_unlink`

# 消息队列

消息队列是 Linux 内核维护的消息链表数组。适合在嵌入式系统中做命令分发和短小数据通信。不推荐使用一个消息队列进行双向通信

整个过程就是先创建一个消息队列，然后发生方把需要发送的消息一条一条的入队，接收方按队列的结构一条一条的出队。但是整个过程还是由内核维护。

## 相关系统调用

### mq_open

`mqd_t mq_open(char *name, int oflag, mode_t mode, struct mq_attr *attr);`

参数1为文件名，命名规则与共享内存一样；参数2为打开模式；参数3为权限；参数4为消息队列的属性信息

返回值为整型，类似文件描述符。

- 作用是创建一个消息队列，返回一个消息队列描述符。位置在`/dev/mqueue`
- 创建方创建消息队列后，另一方打开这个队列也是用`mq_open`，但是不填参数3和4
- `struct mq_attr`：用于描述消息队列的各种属性，在创建消息队列时要先对它赋值初始化。

### mq_timedsend

`int mq_timedsend(mqd_t, char *, size_t, unsigend int, struct timespec *)`

参数1为消息队列描述符；参数2为指向消息的指针；参数3为指向的消息的长度；参数4为指定消息的优先级；参数5为指定等待的最晚时间。

- 作用是发送消息，本质内核是将消息入队
- 消息的顺序是按队列的结构，高优先级排前面。优先级一样新的排后面
- `struct timespec`是一个时刻，而非时间长度，即等到某一时刻队列还是满的就会返回错误

### mq_timedreceive

`ssize_t  mq_timedreceive(mqd_t, char *, size_t, unsigend int, struct timespec *)`

参数1为消息队列描述符；参数2用于接收消息；参数3为用于接收消息的长度；参数4为记录接收到的消息的优先级；参数5为指定等待的最晚时间。

- 作用是按队列的结构，取走入队时间最早权限最高的消息，本质是内核将消息出队

### mq_unlink

`int mq_unlink(char *)`

参数为消息队列名

- 用于清除对应的消息队列，对应文件系统里的文件会被清除

### clock_gettime

`int clock_gettime(clockid_t clk_id, struct timespec *tp)`

参数1为指定用哪个时钟；参数2用于接收转换的时间，`mq_timedsend`和`mq_timedreceive`会用到

- 作用是获取指定时钟的当前时间，并转换成`struct timesoec`类型，精确到ns。

## 实现流程

1. 创建消息队列属性结构体并初始化，再使用`mq_open`创建消息队列
2. 使用`mq_timedsend`将消息入队；另一进程使用`mq_timedreceive`将消息出队
3. 两个进程都关闭消息队列描述符
4. 创建方执行`mq_unlink`清除消息队列

# 信号

- 信号是由内核、其他进程或命令行工具发送给目标进程的，告诉进程发生了某件事（类似中断），进程去处理完后回来继续执行自己的代码。每种信号都用一个唯一的整数值来表示。
- 每个进程的内核控制块都有一张**信号处置表**，本质是一个函数指针数组，以信号的编号作为索引。当进程接收到某个信号时，内核会查看进程的信号处置表，根据接收到的信号编号找到要执行的对应函数。
- 处置表里指向的函数默认在内核空间，由内核写死
- `SIG_IGN`即接收到信号时直接忽略什么都不做；`SIG_DFL`即接收到信号时去内核执行默认的函数

## 常见信号

- `SIGINT`(2)：用户按下crtl+c时发送给前台进程的信号，用于请求进程终止
- `SIGKILL`(9)：强制终止进程的信号，无法捕获、忽略、自定义
- `SIGTERM`(15)：由管理员或其他进程发送，用于请求进程终止
- `SIGUSR1`(10)、`SIGUSR2`(12)：用户自定义信号

## 自定义信号处理函数

通过编写`signal`函数，使信号处置表里对应信号指向的函数变成`signal`，使进程在接收到某一信号时去调用这个函数而不是去调用内核里默认的。

### 相关系统调用

`sighandler_t signal(int signum, sighandler_t handler)`

参数1为要处理的信号；参数2为接收到对应信号时要执行的函数；成功返回该信号对应的旧的信号处理函数

- 调用该函数后，进程就会在接收到`signum`时去运行`handler`函数，即改变信号处置表的内容
- `sighandler_t`可以是自定义函数、`SIG_DFL`或`SIG_IGN`

`int kill(pid_t pid, int sig)`

参数1为要发送给哪个进程；参数2为要发送的信号编号

- 主要作用是给某个进程发送一个信号
